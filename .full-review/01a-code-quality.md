## High

### H-1 — `manifest/free.ts` is hand-authored, not generated

**File:** `packages/medieval-hexagon-gameboard/src/manifest/free.ts` (16,561 LOC, 28% of the entire library)
**Symptom:** Header is a plain JSDoc with `@module` — no `// AUTOGENERATED — DO NOT EDIT` banner, no companion generator script that emits this file. The `ingest.ts`/`generateManifestFromSource` pipeline writes a JSON file consumers can use; this `.ts` file is a separately committed TypeScript literal of the same data.

```ts
/**
 * Packaged FREE KayKit Medieval Hexagon manifest exported as typed runtime data.
 * @module
 */
import type { MedievalHexagonManifest } from '../types';

export const freeManifest: MedievalHexagonManifest = {
  "schemaVersion": "1.0.0",
  "generatedAt": "2026-05-22T00:00:00.000Z",
  ...
```

**Why it matters:**
1. Any change to manifest shape requires hand-editing 16k lines of object literal — drift between schema and bundled data is almost guaranteed.
2. The `"generatedAt"` timestamp is a fiction (frozen string, not a build artefact).
3. Tree-shaking is unlikely to drop unused branches of this giant literal — every consumer of *anything* in the package pays the parse+memory cost.
4. It dominates `tsup` build time and IDE responsiveness; any single-line edit dirties an 800KB file in the diff.

**Fix recommendation:**
- Treat `assets/free/manifest.json` as source of truth; have `tsup` or a `prebuild` script generate `manifest/free.ts` mechanically (`writeManifestModule()` from `ingest.ts` looks like it already does this — wire it into the build).
- Add a literal banner: `// AUTOGENERATED FROM assets/free/manifest.json — DO NOT EDIT BY HAND`.
- Add a CI check (`scripts/audit-package.ts` is a good home) that regenerates and `git diff --exit-code`s.
- Consider exporting from JSON via `import freeManifest from '../assets/free/manifest.json' assert { type: 'json' };` — eliminates the duplicate-of-truth entirely. Node ≥22 supports import attributes; TS5 supports them too.

### H-2 — `cli.ts main()` is a 463-line if/else dispatcher (CC ~854 across 296 functions in one file)

**File:** `packages/medieval-hexagon-gameboard/src/cli.ts:230` (function `main`, 463 lines)
**Other hotspots in the same file:** `usage` (153 lines, `cli.ts:4137`), `runGuideScenarios` (132 lines, `:1810`), `runBlueprint` (101 lines, `:952`), `runGuideUsages` (97 lines, `:1944`). Whole file is 4,297 LOC with cyclomatic complexity ~854 (362 `if`s, 70 `throw`s, no `switch` at all).

The dispatch is a flat sequence:

```ts
function main(argv: string[]): void {
  const parsed = parseArgs(argv);
  ...
  if (parsed.command === 'doctor') { ... return; }
  if (parsed.command === 'validate') { ... return; }
  if (parsed.command === 'manifest') { ... return; }
  if (parsed.command === 'validate-manifest') { ... return; }
  if (parsed.command === 'analyze') { ... return; }
  if (parsed.command === 'declarations') { ... return; }
  // ...30+ more commands...
}
```

**Why it matters:**
- ~30 command branches. Adding a command requires editing `main()`, `usage()`, and `parseArgs` flag handling in three different places — easy to forget one.
- 55 `process.exit(1)` scattered across `cli.ts` — failure modes are coupled to the CLI entrypoint, making it hard to unit-test command handlers (you can't import them without risking exit calls).
- 70 `throw`s as user-facing error reporting — each command bespoke-validates its own flags (`'compatibility requires --asset <path>'`, `'piece requires --asset <path>'`, etc.).

**Fix recommendation:** replace the if-chain with a command registry.

```ts
interface Command {
  name: string;
  requiredFlags?: string[];
  run: (parsed: ParsedArgs, ctx: CliContext) => Promise<void> | void;
  usage: string;
}

const COMMANDS: ReadonlyArray<Command> = [
  { name: 'doctor',   run: (a, c) => a.flags.coverage ? runCoverage(a) : runDoctor(c.sourceRoot, c.edition), usage: '...' },
  { name: 'compatibility', requiredFlags: ['asset'], run: runCompatibility, usage: '...' },
  ...
];

function main(argv: string[]): void {
  const parsed = parseArgs(argv);
  const cmd = COMMANDS.find(c => c.name === parsed.command);
  if (!cmd) { usage(1); return; }
  validateRequiredFlags(parsed, cmd.requiredFlags);  // single point for the 'requires --x' throws
  cmd.run(parsed, { sourceRoot, edition });
}
```

Benefits: `usage()` is derived from the registry (one source of truth), each `runX` handler becomes individually testable without `process.exit`, the "X requires --flag" boilerplate centralizes, and adding a command is one entry instead of three edits.

### H-3 — `simulation.ts` has two parallel `switch (step.action)` blocks for the same enum (~10 cases each)

**File:** `packages/medieval-hexagon-gameboard/src/simulation.ts`
**Locations:**
- `simulation.ts:1643` — `switch (step.action)` (10 cases)
- `simulation.ts:3795` — `switch (step.action)` (10 cases, same enum)
- `simulation.ts:3955` — `switch (effect.type)` (related effect dispatch)

The two action-switches are doing related work on the same discriminated union (`'actor-target-command' | 'command' | 'inspect-actor-targets' | 'remove-actor' | 'remove-placement' | 'spawn-actor' | 'spawn-placement' | 'update-actor' | 'update-placement' | 'run-systems'`).

**Why it matters:**
- Adding a new `step.action` variant means updating two switches. If TypeScript can't prove exhaustiveness (e.g. no `never`-returning `default`), one will silently fall through.
- 5,213 LOC for one module with two near-duplicate dispatch tables is a strong SRP smell: this module is doing both *script interpretation* and *script summarisation/reporting*.

**Fix recommendation:** centralize the action→handler dispatch in one map. Each action gets one entry that both the executor and the summarizer key off:

```ts
type ActionDispatch = {
  [K in GameboardSimulationStep['action']]: {
    execute: (step: Extract<GameboardSimulationStep, { action: K }>, world: World) => StepResult;
    summarize: (step: Extract<GameboardSimulationStep, { action: K }>) => StepSummary;
  };
};

const ACTION_HANDLERS: ActionDispatch = {
  'spawn-actor':   { execute: executeSpawnActor,   summarize: summarizeSpawnActor },
  'remove-actor':  { execute: executeRemoveActor,  summarize: summarizeRemoveActor },
  ...
};
```

Add an exhaustiveness assertion (`function assertNever(x: never): never`) at the call site so the compiler catches missing variants.

This is also the right point to consider splitting `simulation.ts` into `simulation/run.ts`, `simulation/script.ts`, `simulation/summarize.ts`, and `simulation/violations.ts`. 5k LOC in one file passes the "one author can hold it in head" gate only if it's a config table — it isn't here.

## Medium

### M-1 — Audit scripts duplicate `readJson` / `readRequired` / `workspaceRoot` boilerplate

**Files (the same ~5-line helpers reimplemented):**

| Helper | Locations |
|---|---|
| `function readJson<T>` | `scripts/audit-api-docs.ts:130`, `scripts/audit-free-assets.ts:234`, `scripts/audit-workspace.ts:1155`, `scripts/smoke-built-cli.ts:1179` |
| `function readRequired` | `scripts/audit-api-docs.ts:134`, `scripts/audit-reference-assets.ts:592`, `scripts/audit-workflows.ts:139`, `scripts/audit-workspace.ts:1163` |
| `const workspaceRoot = resolve(import.meta.dirname, '..')` | 10 scripts |
| `const packageRoot = join(workspaceRoot, 'packages/medieval-hexagon-gameboard')` | 5 scripts |

Each `readJson` has slightly different semantics — `audit-workspace`'s version handles YAML inline, `audit-api-docs`'s composes with `readRequired`, `audit-free-assets`'s does neither. That subtle drift is the actual bug risk: a fix in one (say, a tolerant JSON parser, or a `git ls-files` membership check) won't propagate.

**Fix recommendation:** extract `scripts/_lib.ts` (or `scripts/common.ts`):

```ts
// scripts/_lib.ts
export const workspaceRoot = resolve(import.meta.dirname, '..');
export const packageRoot = join(workspaceRoot, 'packages/medieval-hexagon-gameboard');

export function readRequired(relativeOrAbsolute: string): string {
  const resolved = isAbsolute(relativeOrAbsolute) ? relativeOrAbsolute : join(workspaceRoot, relativeOrAbsolute);
  if (!existsSync(resolved)) throw new Error(`missing required file: ${relativeOrAbsolute}`);
  return readFileSync(resolved, 'utf8');
}

export function readJson<T>(path: string): T {
  const source = readRequired(path);
  if (path.endsWith('.yaml') || path.endsWith('.yml')) return parseYaml(source) as T;
  return JSON.parse(source) as T;
}
```

Then every audit script becomes a few imports + the actual audit logic.

### M-3 — `smoke-packed-consumer.ts` is a 2,489-line script with a single top-level `try` block

**File:** `scripts/smoke-packed-consumer.ts`
The top-level `try { ... }` runs from line 18 to roughly line 535 (the analysis spotted line 2139 as the `} catch`, but inspection shows ~500 lines of imperative setup followed by ~2000 lines of type/runtime assertions like `const ingestSourceRoot: string = defaultSourceRoot('free', '/packed-consumer');` — these read like a compile-time API surface attestation, not a runtime test).

**Issues:**
- The intent of those 2000 lines (line ~536 onwards) is to assert the published package's type surface, but they're inlined into a script that *also* does the `npm pack` + install dance. The two concerns should not share a file.
- Single top-level `try`/`catch` means **any** failure dumps a single error — diagnosis requires guessing which of the 30+ phases failed. Each phase needs its own labeled try or a per-step harness.
- 53 `throw new Error` calls in one script.

**Fix recommendation:**
1. Split into `scripts/smoke-packed-consumer/pack-install.ts` (the `npm pack` + install + CLI exercise) and `scripts/smoke-packed-consumer/types.ts` (the compile-time API surface attestation — that one should just be type-checked, never executed).
2. The runtime portion should be a small harness that runs **labelled phases**:
```ts
const phases: Array<{ label: string; run: () => void }> = [
  { label: 'npm pack',          run: doPack },
  { label: 'install in app',    run: doInstall },
  { label: 'cli: guide-usages', run: () => runCli('guide-usages') },
  ...
];
for (const { label, run } of phases) {
  try { run(); console.log(`OK  ${label}`); }
  catch (e) { console.error(`FAIL ${label}:`, e); process.exit(1); }
}
```

### M-6 — Library calls `process.exit(1)` from 10+ call sites

**File:** `packages/medieval-hexagon-gameboard/src/cli.ts` (lines 364, 412, 482, 526, 529, 569, 572, 650, 657, 660, …)
**Count:** 55 occurrences of `process.exit` inside library source (all in `cli.ts`).

This is acceptable for a CLI entry point but locks each handler to "must run in a process I'm allowed to kill". Means handlers can't be reused programmatically (e.g. by tests, or by a user wanting to embed CLI behaviour in their build pipeline).

**Fix recommendation:** handlers should `throw` a typed `CliError(code, message)`; the single top-level `try/catch` at `cli.ts:4294` already exists — extend it to read `error.exitCode` and exit accordingly. Body of every handler then becomes pure / testable.

```ts
class CliError extends Error {
  constructor(public exitCode: number, message: string) { super(message); }
}
// handler:
if (report.errors.length > 0) throw new CliError(1, formatReportErrors(report));
// top-level:
} catch (error) {
  console.error(error instanceof Error ? error.message : String(error));
  process.exit(error instanceof CliError ? error.exitCode : 1);
}
```

**Files:**

- `packages/medieval-hexagon-gameboard/src/cli.ts:2797` — `return options as unknown as MedievalGameboardBlueprintScenarioOptions;`
- `packages/medieval-hexagon-gameboard/src/cli.ts:2920` — `return payload as unknown as GameboardScenarioSimulationScript;`
- `packages/medieval-hexagon-gameboard/src/simulation.ts:1198` — `const rawScript = script as unknown;` (this one's actually narrowing, fine).
- `scripts/smoke-packed-consumer.ts:1022–1023` — `undefined as unknown as ReturnType<typeof useGameboardRuntime<...>>;` (compile-time type witness, fine in a smoke test).

**Fix recommendation:** the two `cli.ts` cases (2797, 2920) should be backed by `validateBlueprintScenarioOptions(options)` and `validateSimulationScript(payload)` calls that runtime-check the shape and *return* the typed value — the cast then disappears. Each remaining cast should carry an inline comment explaining why no validator is appropriate.

### L-2 — Most `catch` blocks discard the error context

**File:** `packages/medieval-hexagon-gameboard/src/interop.ts:2292`, `scenario.ts:1163,1375`, plus a few others use bare `catch {}` to convert "parseHexKey failed" into `undefined`.

```ts
try {
  return parseHexKey(target);
} catch {
  return undefined;
}
```

This is fine when the consumer expects "input was not a hex key, treat as absent". But there's no `parseHexKey` boolean variant — every caller pays a `throw/catch` for invalid input on the hot path.

**Fix recommendation:** add `tryParseHexKey(s: string): HexCoordinates | undefined` to `coordinates.ts`; remove every `try { parseHexKey } catch { undefined }` site. Same applies to similar non-throwing variants where flow-control-via-exception is in evidence.

## Top 5 priorities

1. **H-1: Auto-generate `manifest/free.ts` from `assets/free/manifest.json`.** 16,561 hand-maintained lines is the single biggest correctness risk in the package and a meaningful publish-size/parse-cost burden. One `prebuild` script + a CI drift check resolves it.
2. **H-2: Extract a command registry in `cli.ts`.** Collapses `main()` (463 lines), `usage()` (153 lines), and 30+ ad-hoc flag-validation throws into one declarative table. Unlocks unit-testability of every `runX` handler.
3. **H-3: Unify the two `step.action` dispatches in `simulation.ts` and split the file.** A 5,213-LOC module with two parallel switches over the same union is a maintainability bomb — `assertNever` exhaustiveness checks would also catch a class of bugs the test suite probably misses today.
4. **M-1 + M-2: Extract `scripts/_lib.ts` and split `audit-workspace.ts`.** ~10× shared boilerplate across audit scripts and an 89-function god-script erode the safety of the audit harness itself. Fix this before adding more audits.
5. **M-3: Decompose `smoke-packed-consumer.ts`.** Separate the runtime smoke (npm pack + install) from the compile-time API attestation (the 2000 lines after line ~535), add labelled-phase harness so failures point at the right step. A single 2,489-line `try` block is hostile to anyone diagnosing a CI break.

The library is unusually clean for its size — fixing the five items above (none of which is a correctness bug today) lifts the package from "well-disciplined initial scaffold" to "easy to maintain in steady state".
