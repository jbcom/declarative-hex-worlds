## Findings by Severity

### Critical

**C1 — No install / quickstart path for consumers**

`README.md` has only four headings: intro paragraph block, License, Documentation, Development. There is no:
- `npm install @jbcom/medieval-hexagon-gameboard`
- A minimal working code example (create board → render)
- Explanation of which subpath to import first

A new consumer who lands on the README cannot start. This is the first thing any 1.0 library must have.

Outline for `README.md` install section:
```
## Install
npm install @jbcom/medieval-hexagon-gameboard

## Quickstart — live runtime (Koota)
```

130+ raw `throw new Error(...)` calls across the codebase with no grouping. Error messages are meaningful strings (e.g. `"validate-plan requires --plan <path>"`, `"blueprint --outScenario requires scenario options"`) but are not documented as a catalog. Consumers catching errors from CLI subprocess or library calls cannot write robust error handling without grepping source.

**H3 — Determinism contract not documented for consumers**

Seed/reproducibility is extensively used internally: `seed` params on navigation, blueprint, spawn, layout fill, patrol, piece placement. `seedrandom` is the engine. `dist/types.d.ts` says "Stable generation timestamp used by reproducible package assets." But there is no guide explaining:
- What is guaranteed to be reproducible given the same seed across versions
- What breaks reproducibility (engine upgrades, schema version bumps, adding new terrain types)
- Which APIs are seed-stable vs. seed-influenced-but-not-stable

This is a correctness contract for any game that saves/loads boards. Missing at 1.0 is High severity.

Outline — add `docs/guides/determinism.md`:
```
# Determinism and Reproducibility
## What is guaranteed
  - Same seed + same schema version + same board shape → identical GameboardPlan
  - Asset variant assignment is seed-stable
## What breaks reproducibility
  - Schema version bumps (MEDIEVAL_HEXAGON_SCHEMA_VERSION changes)
  - New terrain/asset types added to generation pipeline
  - seedrandom library major version change
## Seed format
  - string | number — normalized to string internally
  - Recommend: game-scoped prefix ("my-game:<mapId>") to avoid cross-system collisions
## Saving and loading
  - GameboardPlan is fully serializable — store as JSON
  - Do NOT store the seed alone; store the full plan for saved games
## Per-API seed behavior
  - createSeededGameboardPlan: top-level seed controls all sub-generation
  - navigation spawn seed: scoped to that call; independent of plan seed
  - patrol-routes seed: per route-set, not inherited from plan
```

**H4 — Peer dependency usage guidance absent**

`peerDependencies`: `react >=18.0.0 <20`, `@types/react`, `three >=0.180.0 <1`. The `./react` and `./three` subpaths require these. The umbrella `.` export does not. But this is documented only in code comments and AGENTS.md (maintainer-facing). Consumers need to know:
- `react` and `three` are optional — do not install if not using React/Three subpaths
- Importing `./react` without react installed gives a cryptic "cannot find module" error at runtime
- TypeScript: `skipLibCheck` or explicit `@types/three` install needed in consuming projects

Outline — add "Peer Dependencies" section to README and `docs/guides/public-api.md`:
```
## Peer Dependencies
This package has optional peer dependencies.
- **react + @types/react** — required only if you import from `./react`
- **three** — required only if you import from `./three`
- The main package (`.`, `./gameboard`, `./runtime`, etc.) has NO peer dependency requirements.
```

**H5 — Trait identity / splitting hazard undocumented for consumers**

`tsup.config.ts` has `splitting: true`. Koota trait identity depends on object reference equality — if `splitting: true` causes the same trait to be bundled into two chunks, trait queries silently return empty. This hazard is known to the maintainer (AGENTS.md) but appears nowhere in consumer-facing docs. Any consumer who bundles this library into their own build with code splitting enabled may hit the same issue without knowing it.

Outline — add warning to `docs/guides/public-api.md` and `docs/pillars/05-koota-runtime-rules.md`:
```
## Bundler Warning: Trait Identity
Koota traits use object reference equality for entity queries. If your bundler
splits the medieval-hexagon-gameboard chunks such that a trait module is
instantiated twice, trait queries will silently return nothing.

Mitigation:
- Import from the pre-bundled dist (default CJS/ESM exports) — do not re-bundle
  the library source yourself
- If you must bundle: ensure medieval-hexagon-gameboard is in `external` or
  `noSplit` in your bundler config
- Test: `world.query(TileTrait)` should return the same count as the plan's
  tile count immediately after createGameboardRuntimeFromScenario
```

### Medium

**M1 — Trait taxonomy has no umbrella index**

Koota traits/relations are spread across 6+ modules (`koota`, `actors`, `movement`, `patrol`, `quests`, `systems`). No single `docs/api/traits.md` or guide section lists all traits with: name, type (component / relation / tag), what entities they attach to, what world queries them, what actions mutate it. Consumers building custom systems need this to avoid re-implementing existing traits.

**M2 — TSDoc density uneven**

`runtime.ts` has dense per-field `/** */` comments on all interface members — good. The barrel `index.ts` has zero TSDoc (type-only re-exports, acceptable). But the 41 subpath modules should each have a `@module` top-level JSDoc so TypeDoc generates a meaningful module-level description. README states "every TypeDoc entry point must carry top-level `@module` JSDoc" — confirm compliance is enforced by `pnpm test:api-docs`.

**M3 — Examples are JSON files, not runnable code**

`docs/examples/` has 6 JSON scenario/recipe files. These are good reference artifacts. But there are no runnable TypeScript examples in docs showing: create plan → apply recipe → start runtime → run frame loop → read actor state. The `./examples/simple-rpg-usage` and `./examples/blueprint-board-usage` subpath exports exist but their contents are only discoverable by reading source or TypeDoc HTML.

**M4 — AGENTS.md is 49KB of maintainer rules**

AGENTS.md is thorough and well-structured for AI agents/maintainers. But it is not a contribution guide. `CONTRIBUTING.md` and `STANDARDS.md` are absent. New human contributors have no clear path.

**M5 — No docs/guides/cli.md (distinct from C2 above)**

`docs/guides/release-readiness.md` is auto-generated. The six human-authored guides cover: public API, guide coverage, runtime integration, recipes/scenarios, rendering/assets. CLI is conspicuously absent given 4,297 LOC and 30+ commands.
