# PRD ‚Äî `@jbcom/medieval-hexagon-gameboard` 1.0

> Authoritative scope for the v1.0.0 release. Folds in every finding from `.full-review/` (Phase 1-5) and gaps surfaced by the `standard-repo` profile audit. **Execution queue** lives in `.agent-state/directive.md` ‚Äî this PRD is the *why*, the directive is the *what next*.

---

## 1. Product statement

`@jbcom/medieval-hexagon-gameboard` is a **deterministic, published-to-npm TypeScript library** for building turn-based games on KayKit's Medieval Hexagon asset pack. It ships:

- A Koota-ECS state model (traits, actions, queries)
- A hex-grid runtime built on `honeycomb-grid`, deterministic via `seedrandom`
- A scenario / blueprint / simulation engine
- A CLI (`medieval-hexagon-gameboard`) for headless validation, simulation, coverage, and asset ingest
- First-class React + Three.js bindings (hard runtime `dependencies`, not optional peers)
- A pre-baked FREE-edition KayKit asset manifest

1.0 is the **first public release**. Beyond this point, semver applies to every supported export, every CLI flag, and the determinism contract.

## 2a. Asset model: bootstrap, not bundle (2026-05-26 correction)

The library does NOT ship the KayKit asset tree. The tarball ships:

- The TypeScript runtime (`dist/`).
- The CLI (`dist/cli.js`).
- The **manifest as JSON metadata** ‚Äî what the consumer's asset tree should look like after bootstrap.
- Schemas / loaders / validators.

Asset acquisition is consumer-driven via the CLI's **`bootstrap`** subcommand (and a programmatic mirror, `bootstrapKayKitAssets({...})`). Behaviors:

1. **Default target**: `public/assets/models/` ‚Äî heuristically detected (`public/assets/`, `public/`, `static/`, `assets/`, `<cwd>`) or explicit via `--out <dir>`.
2. **Source selection**:
   - `--source github` (default for FREE): clones / sparse-checkouts `https://github.com/KayKit-Game-Assets/KayKit-Medieval-Hexagon-Pack-1.0.git`, extracts only the `Assets/gltf/` subtree.
   - `--zip <path>`: accepts a zip file (FREE or EXTRA from itch.io); auto-detects edition by inspecting the archive's manifest/marker files.
3. **Layout**: mirrors upstream structure exactly under `<out>/addons/kaykit_medieval_hexagon_pack/Assets/gltf/...`. The library reads addons via that path at runtime.
4. **Filter**: gltf files only. Skip `*.fbx`, `*.obj`, `*.mtl`, `*.fbm/`, `__MACOSX/`. Optional `--include-source-formats` opt-in for power users.
5. **Integrity**: write `<out>/addons/kaykit_medieval_hexagon_pack/.bootstrap.json` capturing source URL/zip, SHA256 of each extracted file, total count, edition, library version, and a timestamp. `bootstrap --verify` re-checks integrity without re-downloading.
6. **Idempotency**: `bootstrap` is safe to re-run; existing files are checksum-compared, drift produces a clear diff report; `--force` re-fetches.
7. **Determinism**: same source (commit SHA / zip checksum) ‚áí same tree, byte-identical.

Tests exercise both shapes:
- **Unit**: zip parsing, gltf filtering, layout mirroring, integrity report shape (no network).
- **Integration**: end-to-end with `references/KayKit_Medieval_Hexagon_Pack_1.0_FREE/` fed through `--zip` path.
- **E2E**: full `bootstrap` ‚Üí runtime asset resolution ‚Üí render path, both `--zip` and `--source github` (the GitHub variant runs in scheduled CI only ‚Äî not on every PR ‚Äî to bound rate).

## 3. Use cases for the runtime

This is the use-case enumeration step (per CLAUDE.md). Every architectural decision in the queue is anchored here.

| Use case | Consumer | Entry point | Latency budget | Identity-stability contract |
|---|---|---|---|---|
| **Headless scenario validation** | CI scripts, content authors | CLI `validate` / `validate-manifest` | ‚â§ 100 ms warm | Pure function over scenario JSON |
| **Headless coverage ledger** | CI scripts, release-please | CLI `coverage` (programmatic + JSON output) | ‚â§ 200 ms | Deterministic given fixed `--generatedAt` |
| **Headless simulation replay** | CI, save/load systems | CLI `simulate` + library `runSimulationStep` | ‚â§ 1 ¬µs/step for sim-only workloads | Same seed + same script ‚áí byte-identical output across processes |
| **Browser gameplay (React)** | Game frontends | `@jbcom/medieval-hexagon-gameboard/react` + umbrella traits | per-frame budget 16 ms; selector hook ‚â§ 0.5 ms | Trait identity stable across subpaths (`splitting: true` invariant) |
| **Browser gameplay (Three)** | Custom Three.js renderers | `@jbcom/medieval-hexagon-gameboard/three` + umbrella | per-frame 16 ms | Same data ‚áí same GLTF URL set |
| **Asset ingestion** | Content tooling | CLI `extract` / `manifest` / library `ingest.ts` | one-shot, latency-bound by I/O | Same source tree ‚áí same manifest JSON (mod symlinks) |
| **Library-of-library embed** | Downstream npm packages | umbrella `'@jbcom/medieval-hexagon-gameboard'` | parse ‚â§ 30 ms | Tiered public API; semver-safe deprecations |

Tier rules that fall out of this matrix:

- **Headless use cases must NOT pay browser-bundling costs** ‚Üí manifest-as-JSON, lazy CLI subcommands.
- **Browser use cases must NOT pay manifest cost unless they ask for it** ‚Üí `freeManifest` off the umbrella.
- **All use cases share the determinism contract** ‚Üí one cross-process test pins it.

### Epic R ‚Äî Restructure (precedes everything else)

The monorepo shape is wrong. The library is a **single package** that should live at the repo root, with `src/` decomposed into **domain sub-packages** that talk to each other only through barrel public APIs. SimpleRPG is **not** consumer-facing ‚Äî it's a test fixture for integration + e2e. Three deliverables:

1. **De-monorepo.** Delete `pnpm-workspace.yaml`; move `{src,tests,*.config.ts,tsconfig.json,tsup.config.ts}` to repo root. Merge `package.json` files (single `package.json` at root with the `@jbcom/medieval-hexagon-gameboard` name + dependencies + exports + scripts). Drop Nx (we have one package, no graph to coordinate). Drop `apps/docs/` workspace entry ‚Äî keep `docs/` content + vitepress config but don't register as a workspace package.
2. **Decompose `src/`.** Today's 38 flat modules collapse into ~13 domain sub-packages, each with its own `index.ts` barrel and TSDoc-tagged public surface. Cross-domain imports MUST traverse the barrel, never reach into a sibling's files. Biome `noRestrictedImports` enforces. See Appendix C for the target layout.
3. **Relocate SimpleRPG to tests.** `examples/simple-rpg-usage.ts` (and the `examples/*.json` SimpleRPG fixtures) move into `tests/integration/simple-rpg/` (unit-level fixture + assertions) and `tests/e2e/simple-rpg/` (playwright happy-path + failure cases). The bundled examples published in 0.x are removed from the `exports` map; the `examples` directory in the published tarball drops to `examples/blueprint-board-usage` only (or, more likely, also moves to tests).

This block precedes Epic A-G because every downstream epic addresses paths that change shape. After Epic R lands, the directive items in Epics B-G are re-pointed at the new paths.

Notes vs the earlier draft:

- **No `koota/` sub-package.** That conflated *the dep `koota`* with *our integration of it*. Koota is consumed; the integration shape it expects (traits/, systems/, actions.ts, world.ts) is what we mirror.
- **`runtime/` flattened.** Coordinate work splits naturally into `coordinates/`; gameboard/occupancy/navigation belong together under `gameboard/`; pieces get their own home; nothing left over for a "runtime" name.
- **`actors/` ‚Üí `traits/actors.ts` + `systems/movement-system.ts` etc.** Per koota conventions, traits are flat declarations; systems read traits and act. Splitting along subsystem (movement, patrol, quests) and concern (trait file vs system file) matches how koota itself documents app structure.
- **`commands.ts` + `selectors.ts` are `@internal`** ‚Äî they're plumbing for `actions.ts` and React hooks, not public surface.
- **`world.ts`, `actions.ts`, `frameloop.ts`, `startup.ts`** sit at the `src/` root because they're the composition layer ‚Äî they orchestrate everything else and need to see every sub-package.

**Public-surface rule (Biome `noRestrictedImports`):** within `src/`, the only allowed import path from another sub-package is `'./<sibling-name>'` (the barrel). Never `'./<sibling-name>/some-internal.js'`. Test files in `tests/` may reach internals **only** through documented `tests/internal/` re-exports so the contract is explicit.
